Session A2+A3-lite — Data Pipeline (EURUSD, XAUUSD, BOOM500)

Implementation plan · 2026-06-12 · LSB Strategy Bot v3.0

1 Overview

Build a fetch → audit → load pipeline that gets ≥3 years of audited H1 OHLCV for three instruments (EURUSD, XAUUSD, BOOM500) into the candle table, so Sessions A4–A9 have real data to evaluate gates and run walk-forward on. Combines the originally-separate A2 (instrument #1) and A3 (instruments #2–n) sessions, since the fetch/audit/load scripts are instrument-agnostic via InstrumentConfig.

Each instrument gets its own docs/data_audit_report_<INSTRUMENT>.md with anomalies and written dispositions. A new config/XAUUSD.yaml is added alongside the existing EURUSD/BOOM500 configs.

2 Approach & decisions

DecisionOptionsTrade-offChoice
EURUSD / XAUUSD source Dukascopy tick (.bi5) download, resampled to H1 vs. Stooq H1 CSV export Dukascopy is free, no API key, deep history, but tick-level (heavy parsing: ~1 file/hour/symbol for 3 years ≈ 26k files) and datafeed.dukascopy.com timed out when probed from this sandbox (main site reachable, datafeed subdomain wasn't). Stooq's https://stooq.com/q/d/l/?s=<symbol>&i=60 serves H1 OHLC directly for both eurusd and xauusd, free, no account — but its CSV endpoint now sits behind a small JS proof-of-work (grind a nonce until SHA256(challenge+nonce) has 4 leading hex zeros, POST to /__verify for a cookie, retry). Trivial compute (≈2^16 hashes), ~10 lines. Dukascopy primary, Stooq named as the fallback, both behind a shared fetch_ohlcv(instrument, start, end) interface so either can be swapped in per-instrument without touching the audit/load code. Connectivity to Dukascopy re-checked first at run time; falls back to Stooq (with PoW-solve helper) if unreachable.
BOOM500 source Deriv WebSocket ticks_history (style=candles) vs. manual CSV export Resolved earlier — user has no MT5/cTrader export handy; Deriv's public endpoint needs no auth. Deriv WebSocket, paginated backwards via end timestamps (max ~5000 candles/request → ~7 months of H1; need ~5 requests for 3 years).
Parquet cache location data/history/*.parquet committed vs. gitignored Committing makes audit reproducible without re-fetching, but adds binary churn to the repo; the candle table in Postgres is already the durable record. Gitignore data/history/. The audit report records source, date range, and fetch timestamp so the run is reproducible/re-fetchable.
Load upsert semantics ON CONFLICT (instrument, timeframe, ts) DO NOTHING vs DO UPDATE DO NOTHING is safer against accidental overwrite, but blocks re-loading corrected bars after a disposition fix. DO UPDATE SET open/high/low/close/volume/spread — re-running the loader after correcting a flagged candle fixes it in place.
Audit gate workflow Hard-fail script on any gap>2 vs. report-then-acknowledge A hard fail with no escape hatch can't distinguish "real data bug" from "expected market closure (Christmas, New Year)". audit_history.py always writes the anomaly table into the report. A ## Dispositions section (hand-written, keyed by anomaly id) must cover every flagged unexplained gap>2 before --load is allowed to proceed; otherwise it aborts non-zero.

6 Failure modes

What could go wrongHow the plan handles it
datafeed.dukascopy.com unreachable (observed timeout during planning) fetch_ohlcv() tries Dukascopy first, falls back to fetch_stooq_h1 (Stooq H1 CSV) for EURUSD/XAUUSD if it fails; the audit report records which source was actually used.
Stooq's proof-of-work / cookie mechanism changes shape (anti-bot tweak) PoW-solve is isolated in fetch_stooq_h1's request helper; if it breaks, the symptom is a clear HTTP/parse error on that one function, not a silent bad-data load (audit still runs on whatever Parquet exists).
Deriv ticks_history request cap (~5000 candles) can't cover 3 years in one call Paginate backwards using the end param across ~5 requests; dedupe on timestamp before resample.
3 years of Dukascopy tick files = tens of thousands of small downloads Rate-limited sequential download with on-disk resume (skip files already fetched); only triggered for EURUSD/XAUUSD, not BOOM500.
"Fail on unexplained gap>2" could block load forever on legitimate holiday closures Disposition-gate workflow (Decision table, row 4): every flagged gap needs a written disposition (e.g. "Christmas/New Year close — expected") before --load proceeds.
DST check is meaningless for BOOM500 (Deriv synthetics trade 24/7, no DST) check_dst_anomalies and check_weekend_bars only run when asset_class == "fx" (config-driven, same flag XAUUSD/EURUSD already set).
