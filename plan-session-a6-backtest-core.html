Session A6 — Event-Driven Backtest Core

Implementation plan · 2026-06-18 · LSB Strategy Bot (Phase A, Week 2)

1 Overview

A6 builds the deterministic replay engine that drives the (already-complete) signal engine across history. An injectable Clock streams data/history/<INSTR>_H1.parquet candle-by-candle into signals.engine.evaluate(); qualified signals open a position that is then walked through its lifecycle — pending → filled → managed → closed — per the v2.1 trade-management rules (Requirements §10 order execution, §11 breakeven / trail / defensive exits). Every evaluated candle writes a signal row.

Scope (strict, per Implementation Plan v3.0): IN: Clock · replay loop · position book · §11 management · pyramiding · signal rows OUT: pessimistic fills → A7 OUT: wf_run / sim_trade / walk-forward → A9 OUT: equity / stats / verdict → A10

The fill is the seam: A6 ships a Broker protocol with a naive optimistic fill so the lifecycle runs end-to-end; A7 drops in SimulatedBroker with the Part 7.1 pessimistic model behind the same protocol — no loop changes.

⚠ Pyramiding is net-new strategy logic. Requirements v2.0 §10–§11 and Blueprint v2.1 are written for a single open position — they contain no scale-in / add-on rules. Per the owner's decision (2026-06-18) the bot supports pyramiding from the start, so A6 manages a position book (multiple concurrent legs, same instrument). The add-policy below is a proposed default in standard pyramiding form (add only to winners, each leg independently stopped, capped depth) and should be ratified as ADR-003 — it is not drawn from the existing spec.

src/lsb/backtest/book.py — pyramiding policy over the open legs (net-new, ADR-003)

class PositionBook:
    """All open + pending legs for one instrument. Enforces the add-policy."""
    legs: list[Position]
    def may_add(self, signal: SignalResult, p) -> bool:
        if not p.pyramid_enabled:        return len(self.legs) == 0
        if len(self.open_legs) >= p.pyramid_max_legs:          return False
        if self.open_legs and signal.direction != self.direction: return False  # no hedging
        # add only to a winner: newest open leg must be >= pyramid_add_at_r
        return not self.open_legs or self.newest.r_now >= p.pyramid_add_at_r

src/lsb/backtest/loop.py — the driver that ties it together

def run_backtest(candles, config, config_hash, *, broker=NaiveBroker(),
                 sink=NullSink()) -> BacktestResult:
    window = deque(maxlen=MIN_H1_WINDOW * 2)
    book = PositionBook()                       # all concurrent legs
    for bar in ReplayClock(candles):
        window.append(bar)
        # 1. manage EVERY open/pending leg on this close
        #    (fills, §11 stop moves, stop/target/defensive exits, §10.3 expiry)
        #    §11.4 book-wide exits (opposing sweep / structure invalid) close ALL legs
        # 2. evaluate() the closed bar -> SignalResult -> sink.write(result)
        # 3. if qualified and book.may_add(signal, p): stage a new leg
    return BacktestResult(signals=..., closed_positions=...)

Ordering: manage/fill the prior bar's legs before staging a new leg from the current bar — never fill and manage a leg on the bar that created it (look-ahead guard). Per-leg §11 management is unchanged; pyramiding only adds the book-level may_add gate and book-wide defensive exits.

Every transition is driven by an H1 candle close (§11.1). Intra-candle tick logic is explicitly out of Phase A. Fills resolve through the Broker seam, so A7's pessimistic model changes where a fill lands, not the machine.

Pyramiding (ADR-003): the machine above describes one leg. The PositionBook holds several legs in their own states concurrently; each is independently managed by §11. A new same-direction qualified signal stages an additional PENDING leg only when book.may_add() passes (winner, under pyramid_max_legs). Book-wide §11.4 events (opposing sweep, structure invalidation) close all legs at once.

6 Failure modes

What could go wrongHow the plan handles it
Look-ahead bias — managing/filling a position on the same bar that produced its signal, or using the closing bar's own high/low to decide its fill Strict ordering in loop.py: manage the prior bar's position first, evaluate the just-closed bar second, stage new orders for subsequent bars. Document the contract; integration test asserts entry ts > signal ts.
Determinism leak (dict ordering, float accumulation, set iteration) — would fail the A8 gate that lands right after Build A6 determinism-clean from the start: ordered structures only, no set iteration in output paths, no wall-clock. A8 then just wires the golden-replay, not re-architects.
Scope creep into A7 — implementing spread/slippage/gap fills "while we're here" NaiveBroker is deliberately optimistic and labelled as a stand-in. The Broker protocol is the only contract A7 needs; pessimistic math stays out.
EMA21 trail unavailable early in history (§11.3 fallback) divides by missing data Reuse ema_series; when it returns < needed length, fall back to most-recent swing low/high + 5-pip buffer exactly as §11.3 specifies. Unit-tested both branches.
BTCUSD only has ~1y history — a multi-bar window may underflow MIN_H1_WINDOW Loop skips evaluation (and writes no signal) until the window reaches MIN_H1_WINDOW; engine already guards this. The 1y limit bites at A9's 24-month window, not A6.
Crypto/index pip semantics — ±1 pip entry trigger and ±5 pip trail in §10/§11 assume FX pips All pip offsets go through config.pip_size (already asset-class aware, as in risk.py), never hard-coded.
Position closes exactly on stop and target within the same bar — ambiguous outcome Pessimistic tie-break: assume stop hit first (worst case). Encoded in NaiveBroker/manage and inherited by A7. Documented as a fill convention.
Pyramiding stacks unbounded risk — adding leg after leg compounds drawdown if the trend fakes out Hard cap pyramid_max_legs (default 3) + winner-only gate (pyramid_add_at_r): a leg is only added once the prior leg is already in profit, so a reversal closes new legs near breakeven. Book-wide §11.4 exit flattens all legs on opposing sweep / structure break.
Pyramiding is absent from the spec — silently inventing strategy rules would violate the "never re-type the spec" discipline All pyramiding params live in config with documented defaults and are recorded in ADR-003 as net-new owner decisions, explicitly not from v2.0/v2.1. Walk-forward (A9) will measure the edge with pyramiding on; the verdict (A10) judges it.
Per-leg vs aggregate R confusion — pyramiding tempts an equity/sizing model that belongs to A10 A6 tracks each leg at 1 notional R independently (matches the deferred-P&L decision). No account equity, no aggregate sizing in A6; the book only counts legs and reads each leg's r_now.
